Data Migration Strategy: Option 1 vs Option 2

Two AI agents argued opposing positions. A third acted as judge. The judge's ruling became the confirmed decision.

OutcomeOption 1 — full migration on cutover
The question
How do we handle data when we cut over from Direct Apps (Rails) to Salesforce for AA/EC case management?
Option 1
Move all in-flight cases and 6 years of historical data to Salesforce on cutover day. Direct Apps decommissioned immediately.
Option 2
Let in-flight cases finish in Direct Apps first. Only migrate to Salesforce once they reach a terminal state. Historical data follows separately.
Scope
~100 in-flight cases · ~60,000 historical records · 6-year retention requirement · ~28k students
Agent A — Option 1 Advocate
Argues for full migration on cutover
Position: move everything — in-flight and historical — to Salesforce on day one. Clean break. Single source of truth from launch.
Agent B — Option 2 Advocate
Argues for phased / in-flight first approach
Position: let live cases finish in Direct Apps naturally. Avoid migrating cases mid-flow. Reduce cutover risk by keeping two systems briefly.
Agent C — Judge
Independent evaluator · No prior position
Scores each argument on evidence, feasibility, and risk. Issues a binding ruling with reasoning.
Round 1 — Opening Arguments
A
Option 1
Agent A · Opening

Option 1 is the only coherent choice here. The moment you run two systems in parallel — even briefly — you've created a reporting problem you can't easily undo. Salesforce is being positioned as the single source of truth for both operational case management and the two strategic reports (master + trends). If some cases live in Direct Apps and some in Salesforce at the same time, neither report is complete. ORC can't work cases reliably. Results & Awards can't see ECs. You've degraded the very thing you built Salesforce to replace.

The argument for Option 2 sounds cautious, but it conflates "lower risk at cutover" with "lower overall risk." Keeping Direct Apps alive costs money, requires ongoing support from Dan Jackson's team, and delays decommission. Every week of parallel running is a week of dual maintenance.

Migrate everything on day one. Accept the short-term migration effort. Get to a clean state.

B
Option 2
Agent B · Opening

Option 1 sounds clean in theory. In practice, you are proposing to migrate ~100 live cases mid-process. These aren't static records — they are active workflows. An AA case mid-review, with evidence uploaded and an ORC caseworker mid-notes, gets frozen and re-thawed in a completely different system. The risk of data loss, broken attachments, or corrupted state is real and concentrated at the worst possible moment: launch day.

Option 2 says: let the ~100 in-flight cases finish naturally in Direct Apps. This takes weeks at most — AA cases don't linger indefinitely. Meanwhile, Salesforce goes live for all new submissions. Historical migration can proceed in parallel, at lower risk, without time pressure. You get the same end state but with far less cutover day exposure.

The dual-system period is short. The risk of migrating live cases is not.

Round 2 — Cross-Examination
A
Option 1
Agent A · Challenge to B

Agent B assumes AA cases are simple, linear workflows that "finish naturally." That's not the data. An AA case has up to seven distinct status transitions:

  • Submitted → In ORC Review → Rejected
  • → Refer to Board → Board Rejected
  • → Full Application Review → Refer to New Board
  • → COP Letter → Closed

A case that appears closed can be re-entered at almost any stage — including after a student graduates. There is no clean "this case is done" signal you can trust to leave it in Direct Apps. You'd be maintaining Direct Apps indefinitely for a tail of edge cases with no defined end point.

Option 2's "a few weeks" assumption does not survive contact with reality.

B
Option 2
Agent B · Challenge to A

Agent A's reporting argument is valid but overstated. A well-scoped dual-system period — say, 4 weeks — with a clear freeze on new Direct Apps submissions from cutover day would produce a shrinking pool of legacy cases, not an open-ended commitment. Reporting could flag cases as "legacy" with a simple status field. ORC would know which system to look at by case creation date.

The deeper concern with Option 1 is the 60,000 historical records. These aren't just data rows — they include uploaded evidence documents, email threads, and attachments. Field mapping validation in a test environment is not the same as a successful live migration at scale. What's the rollback plan if the migration fails at 40,000 records?

The risk is asymmetric. A failed migration is worse than a 4-week dual-system period.

Round 3 — Rebuttals
A
Option 1
Agent A · Rebuttal

The "legacy flag" workaround Agent B proposes introduces exactly the complexity it claims to avoid. Now you need: a status field in both systems, a cross-system reporting query, ORC trained on which system to use for which case, and a defined trigger to finally decommission Direct Apps. That's not a 4-week problem — that's an architectural decision that creates months of maintenance overhead.

On rollback: the migration happens to a test environment first (Phase 3 of the 6-phase plan). A failed migration at 40,000 records in test is a test failure — fixable before go-live. The production migration runs only once validation has passed. The rollback concern is addressed by the phased migration plan, not by keeping Direct Apps alive.

Option 1 is harder to execute. It is not riskier in the long run.

B
Option 2
Agent B · Rebuttal

I accept Agent A's point on the 7-stage AA status chain — I had underestimated it. That does complicate the "cases finish naturally" argument. A case at "Refer to Board" stage with an upcoming board date cannot simply be left in Direct Apps while all new board activity moves to Salesforce.

However, my core concern stands: the 60,000 historical records include evidence documents and attachments that are not structured data rows. The field-mapping exercise (SF-3681) is still in progress. Migrating attachments at scale — especially from a Rails file store — is operationally different from migrating structured case fields. I would want explicit confirmation that evidence documents are in scope and validated before committing to Option 1 unconditionally.

I am partially conceding on in-flight cases. I maintain my concern on historical attachment migration.

Judge's Deliberation & Ruling
J
Judge
Agent C · Deliberation

I have reviewed both positions across three rounds. My assessment:

Agent B's strongest argument was the attachment/evidence document migration risk — this is a legitimate technical concern that Option 1's advocates must address explicitly in the migration spec. Agent B was also right to probe the rollback scenario.

Agent A's decisive argument was the 7-stage AA status chain. This fundamentally undermines the Option 2 premise. If cases can re-enter at any stage post-closure — including after graduation — there is no principled cutoff point for "Direct Apps cases are done." Option 2 does not have a defined end state. That is not a minor operational inconvenience; it is a structural problem that would require ongoing engineering support for an indefinite period.

Agent A was also correct that Salesforce as the single source of truth is non-negotiable if the two Salesforce reports (master + trends) are to function from day one. A dual-system reporting setup — even temporarily — degrades the strategic value of the entire migration.

Agent B's partial concession in Round 3 on in-flight cases was telling. Once that position is conceded, Option 2 reduces to: "migrate in-flight cases too, but do historical migration more carefully." That is not a separate option — that is a migration execution concern within Option 1.

✓
Ruling: Option 1
Full migration on cutover — in-flight cases + 6-year historical — to Salesforce

Option 1 is the correct approach. The four decisive reasons, in order of weight:

1
No clean terminal state exists for AA cases. Up to 7 status transitions; cases can re-enter at any stage, including post-graduation. Option 2 has no defined end point — it would require Direct Apps maintenance indefinitely.
2
Salesforce must be the single source of truth from day one. Splitting active cases across two systems makes neither the master report nor the trends report accurate at launch. This undermines the entire migration's strategic purpose.
3
Option 2's "short dual-system period" assumption is false. The complexity of AA case re-entry, cross-system reporting, and ORC training overhead means parallel running would extend well beyond any 4-week estimate.
4
Migration risk is manageable via the phased plan. Test environment validation (Phase 3) addresses the rollback concern. Attachment migration must be explicitly scoped in the migration spec (SF-3628) — this is an action item, not a reason to choose Option 2.

Condition: Evidence document / attachment migration must be explicitly confirmed in scope and validated in the test environment before Phase 6 (live migration) proceeds. This is the one material risk Agent B correctly identified.
